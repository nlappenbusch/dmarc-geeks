Anleitung

Vom Aufsetzen über die Tenant-Einrichtung bis zum täglichen Betrieb.

1. Was tut das Tool?

DMARC-Reports sind tägliche Sammelberichte, die Mailprovider (Google, Microsoft, Yahoo …) an die rua-Adresse in deinem DMARC-DNS-Record schicken. Sie enthalten pro Quell-IP, wie viele Mails in deinem Namen verschickt wurden und ob diese SPF/DKIM bestanden haben.

Roh sind das XML-Dateien — schwer zu lesen und schnell viele. Dieses Tool

  • holt sie automatisch per IMAP aus deinem Postfach (oder du lädst sie hoch)
  • parst .xml, .xml.gz und .zip
  • speichert sie pro Mandant (Tenant) und Domain
  • zeigt Auswertungen: Pass/Fail-Rate, Top-Quellen, Verlauf

2. Deployment

2a. Mit Docker (empfohlen)

Voraussetzung: Docker Desktop oder Docker Engine + Compose v2.

# 1) Keys erzeugen
python scripts/generate_keys.py

# 2) .env aus Vorlage anlegen und Keys eintragen
cp .env.example .env

# 3) Hochfahren (Postgres + App)
docker compose up -d --build

# 4) Logs ansehen (optional)
docker compose logs -f app
Tipp: Beim Update später docker compose pull && docker compose up -d --build. Das DB-Volume dmarc_db bleibt erhalten.

2b. Lokal mit Python (Dev/Test)

Voraussetzung: Python 3.12 +. SQLite reicht für ersten Test.

pip install -r requirements.txt
python scripts/generate_keys.py    # Keys notieren
# .env mit DATABASE_URL=sqlite:///./dmarc.db schreiben
python -m uvicorn app.main:app --port 8000

2c. Erstes Login

Beim ersten Start wird automatisch ein Default-Tenant und ein Superadmin aus den ENV-Variablen angelegt:

  • SUPERADMIN_EMAIL — Login-Adresse
  • SUPERADMIN_PASSWORD — Passwort (sofort nach Login ändern → Mein Account)
  • DEFAULT_TENANT_NAME — Name des ersten Mandanten

Sind später User in der DB, werden diese ENVs ignoriert (kein Reset über .env-Edits).

Wichtig: SECRET_KEY und FERNET_KEY nie verlieren. Verlust des FERNET_KEY macht alle gespeicherten IMAP-Passwörter unbrauchbar.

3. Mandanten (Tenants)

Ein Tenant ist eine logisch isolierte Einheit. User von Tenant A sehen nichts von Tenant B. Jeder Tenant hat:

  • eigene Domains
  • eigene Mailboxen
  • eigene User (Admins + Members)

Mandanten anlegen (als Superadmin)

  1. Login als Superadmin → Menü Tenants
  2. Formular „Neuen Tenant anlegen“:
    • Name — z. B. Kunde GmbH
    • Admin-E-Mail — der erste User dieses Tenants
    • Admin-Passwort — min. 8 Zeichen, an Kunden weitergeben
  3. Anlegen → der Kunde loggt sich mit dieser Mail/Pass ein und sieht nur seine Daten.

User pro Mandant

Tenant-Admins können im Menü Users weitere User in ihrem Tenant anlegen:

  • Member — kann Reports lesen, kein Schreiben
  • Admin — kann zusätzlich Domains, Mailboxen und User verwalten
  • Superadmin — nur über Datenbank (oder beim allerersten Bootstrap)

4. Domain einrichten

Bevor Reports importiert werden, muss die Domain im Tenant registriert sein.

4a. Domain anlegen

  1. Menü Domains → „Neue Domain“ → z. B. meinedomain.de
  2. Du landest auf der Detail-Seite und siehst einen Verifikations-Token.

4b. DNS-Verifikation (optional, aber empfohlen)

Trage in deinem DNS-Provider folgenden TXT-Record ein:

meinedomain.de.   IN TXT  "dmarc-aggregator-verify=DEIN_TOKEN"

Auf der Domain-Detail-Seite „DNS prüfen“ klicken → bei Erfolg wird die Domain als verifiziert markiert. Reine Verifikation hat keinen Einfluss auf den Import — sie ist nur ein Beweis, dass dir die Domain gehört.

4c. DMARC-Record setzen (zentraler Schritt!)

Damit überhaupt Reports kommen, brauchst du einen DMARC-DNS-Record:

_dmarc.meinedomain.de.  IN TXT  "v=DMARC1; p=none; rua=mailto:dmarc@meinedomain.de; pct=100"

Erklärung:

  • p=none — nur reporten, keine Aktion. Mit dem starten!
  • rua=mailto:… — die Mailbox, an die ISPs Reports schicken (das ist die Adresse, die du gleich im Tool als IMAP-Mailbox einträgst)
  • pct=100 — alle Mails reporten
Workflow: Erst 2–4 Wochen mit p=none beobachten. Wenn alle legitimen Quellen passen (grün im Dashboard), schrittweise auf p=quarantine, dann p=reject hochziehen.

5. Mail-Abholung (IMAP)

5a. So funktioniert es

  1. ISPs schicken Reports an dmarc@meinedomain.de (deine rua-Adresse).
  2. Der Aggregator pollt diese Mailbox alle 15 Minuten via IMAP.
  3. Anhänge mit .xml, .xml.gz, .zip werden gelesen, geparst, importiert.
  4. Optional: Verarbeitete Mails wandern in einen anderen Ordner.

5b. Felder erklärt

FeldWas rein gehört
BezeichnungBeliebig, nur für dich. Z.B. dmarc@meinedomain.de
HostDer IMAP-Server. Beispiele unten.
Port993 für SSL/TLS, 143 nur ohne SSL (selten).
SSL/TLSPraktisch immer an. Aus nur, wenn dein Hoster es ausdrücklich vorgibt.

5d. Wenn was nicht klappt

Im Mailboxen-View erscheint bei einer fehlgeschlagenen Abholung „Fehler“. Hover über das Badge zeigt die Meldung. Häufige Ursachen:

  • Auth failed — falsches Passwort, oder kein App-Passwort generiert
  • Connection refused / timeout — Port falsch, SSL-Toggle falsch, Firewall blockiert
  • BAD command — Folder-Name falsch (Groß-/Kleinschreibung beachten, IMAP nutzt manchmal INBOX/Underordner)
  • Keine Reports importiert — Mails sind da, aber der Anhang wird nicht erkannt? Nur .xml/.xml.gz/.zip werden verarbeitet

Tipp: nach dem Speichern auf jetzt prüfen klicken, dann sofort sehen, was passiert.

6. Reports manuell hochladen

Kein IMAP konfiguriert oder historische Reports nachladen?

  1. Menü Upload
  2. Dateien auswählen (Mehrfachauswahl möglich)
  3. Hochladen — Status pro Datei wird angezeigt

Status-Bedeutungen:

  • ok — importiert
  • dup — gleicher Report war schon da (kein Fehler)
  • ignored — die Domain im Report ist im Tenant nicht angelegt. Als Admin: erst Domain hinzufügen, dann erneut hochladen — Admins können das Anlegen auch automatisch beim Upload zulassen.
  • error — Datei kaputt oder kein DMARC-XML

7. HTTP-Upload-API & API-Keys

Reports automatisch reinpushen — z. B. aus Postfix-Pipes, CI-Pipelines, oder anderen Tools.

API-Key erzeugen

  1. Menü API-Keys → Label vergeben → Key erzeugen.
  2. Der Key wird einmal angezeigt im Format dmk_<prefix>.<secret> — sofort kopieren und sicher speichern.
  3. Verloren? Nicht rekonstruierbar — neuen erzeugen, alten widerrufen.

Verwendung

curl -X POST {{ request.base_url }}api/v1/reports \
  -H "Authorization: Bearer dmk_PREFIX.SECRET" \
  -F "files=@report.xml.gz"

Antwort

{
  "results": [
    { "filename": "report.xml.gz", "status": "ok", "message": "imported 12 records", "report_id": 42 }
  ]
}

Status: ok, dup (schon importiert), ignored (Domain im Report nicht für Tenant registriert), error.

Sicherheit: Keys können jederzeit widerrufen werden — laufende Integrationen brechen sofort ab. Über das Audit-Log siehst du, wer wann welche Aktion gemacht hat.

7b. Webhooks

Bekomme HTTP-POSTs, sobald ein Report importiert wird — für SIEM, Slack-Bridges, eigene Pipelines.

  1. Menü Webhooks → URL eintragen.
  2. Bei jedem importierten Report wird POST auf deine URL gefeuert.
  3. Test-Button schickt einen test.ping-Event zur Verifikation.

Payload

{
  "event": "report.imported",
  "tenant_id": 1,
  "report_id": 42,
  "domain": "example.com",
  "records": 12
}

Signatur

Header X-DMARC-Signature = HMAC-SHA256 hex(secret, body). Auf Empfängerseite verifizieren — sonst akzeptierst du gefälschte Events.

# Beispiel (Python)
import hmac, hashlib
expected = hmac.new(secret.encode(), body, hashlib.sha256).hexdigest()
assert expected == request.headers["X-DMARC-Signature"]

7c. DMARC-Generator

Klick dir interaktiv den passenden DMARC-DNS-Record zusammen → /dmarc-generator.

  • Drei Presets: Monitoring · Schrittweise schärfen · Voll geschärft
  • Live-Vorschau aktualisiert sich beim Tippen
  • Validierung warnt bei riskanten Kombinationen (z. B. p=reject ohne rua=)
  • Glossar erklärt Alignment, Pass-Rate-Schwellen, SPF/DKIM-Beziehung

Aus jeder Domain-Detail-Seite verlinkt — die Domain wird automatisch vorausgefüllt.

7d. Advisor & Allowlist

Policy-Advisor

Auf jeder Domain-Detail-Seite siehst du Empfehlungen — datengetrieben:

  • Bereit für p=quarantine — wenn 30-Tage-Pass-Rate > 99 % und aktuell p=none
  • Bereit für p=reject — analog ab p=quarantine
  • Niedrige Pass-Rate — Hinweis, legitime Quellen erst zu klären
  • Pass-Rate fällt — wenn die letzten 7 Tage stark vom 30-Tage-Schnitt abweichen
  • Stale Reports — wenn 5+ Tage keine eingingen

IP-Allowlist

Markiere bekannte legitime Sender (z. B. eigene Mail-Server, Marketing-Provider). In der Top-Quellen-Tabelle erscheint dann ein allow-Badge — keine Verwechslung mit Spoofing.

Schnell hinzufügen via + Allowlist direkt in der Top-Quellen-Tabelle.

Hinweis: Allowlist ist informativ, ändert nichts am DMARC-Empfang. Sie hilft dir und deinem Team, schnell zu sehen, was OK ist und was nicht.

7e. Tags

Bei vielen Domains hilft Gruppierung — z. B. Produktion, Marketing, Kunde-X.

  1. Menü Settings → Tags: Tag mit Name + Farbe anlegen.
  2. Auf der Domain-Detail-Seite klickbar zuweisen.
  3. In der Domain-Liste sichtbar — visueller Anker.

7f. Settings, Branding & Alerts

Branding

In Settings kannst du eine Akzentfarbe pro Tenant setzen. Die ganze UI (Buttons, Links, Highlights) richtet sich danach. Ideal, wenn du verschiedene Mandanten visuell unterscheidest.

Wöchentlicher Digest

Mo 07:00 UTC bekommen die konfigurierten Empfänger eine Zusammenfassung pro Domain (Pass-Rate, Volumen). Ohne Empfänger gehen Mails an alle Tenant-Admins.

Spike-Alerts

Bei jedem Report-Import wird die Fail-Rate der letzten 24 h mit der Vorwoche verglichen. Sprung > Schwelle (Default 10 %) bei min. 100 Nachrichten ⇒ Mail.

SMTP nötig: Digest und Spike-Alerts brauchen SMTP_HOST usw. in .env. Sonst wird stillschweigend nicht gesendet.

7g. Audit-Log

Unter Settings → Audit-Log siehst du die letzten 500 Aktionen deines Tenants:

  • Logins / Logouts (auch fehlgeschlagene)
  • Domain-Anlagen, -Verifikationen, -Löschungen
  • API-Key Erzeugen / Widerrufen / Löschen
  • Webhook-Anlagen
  • Allowlist-Änderungen, Tag-Zuweisungen
  • Settings-Änderungen, Passwort-Resets

Pro Eintrag: Zeit, User, Aktion, Ziel, Details, IP.

8. Dashboard verstehen

Charts: Sender-Mix-Donut, Sender-Stack über die Zeit, Sankey (Sender → Alignment → Disposition), Sparklines pro Domain.

KPIs

  • Nachrichten — Summe aller count-Werte (Anzahl Mails laut ISPs)
  • DMARC pass — SPF oder DKIM aligned
  • DMARC fail — weder noch
  • Quarantine / Reject — was Empfänger laut deinem Policy gemacht haben

Verlauf

Stacked Bar pro Tag, grün = pass, rot = fail. Rote Spitzen sind ein Warnsignal — entweder eine fehlkonfigurierte legitime Quelle oder ein Spoofing-Versuch.

Top Quellen

IPs sortiert nach Volumen. Hostname (reverse DNS) hilft beim Erkennen. Was tun, wenn eine Top-Quelle viele Fails hat?

  • Eigener Server (z. B. mail.meinedomain.de): SPF-Record und DKIM-Selektor prüfen.
  • Externer Dienstleister (Mailchimp, Sendgrid, Microsoft 365 …): in SPF aufnehmen oder dort DKIM signieren lassen.
  • Unbekannte IP: Spoofing? Sobald alle legitimen Quellen passen → DMARC-Policy hochziehen.

9. Sicherheit & Backup

  • In Produktion hinter TLS-Reverse-Proxy (Caddy/Traefik/Nginx) und https_only=True in app/main.py setzen.
  • IMAP-Passwörter werden mit Fernet symmetrisch verschlüsselt — der Schlüssel liegt in FERNET_KEY. Sichern.
  • Backup: Postgres-Volume dmarc_db regelmäßig dumpen, oder bei SQLite einfach dmarc.db kopieren.
  • Reports werden inkl. Roh-XML gespeichert. Wenn DSGVO-Bedenken: in app/ingest.py das raw_xml=-Feld weglassen.
  • Reverse-DNS ist Best-Effort (3 s Timeout). Mit RESOLVE_PTR=false deaktivieren, wenn der DNS-Server nicht erreichbar ist.

10. FAQ & Troubleshooting

Kein Report taucht auf, obwohl ich rua= gesetzt habe.
  • DNS-Propagation kann 1–24 Stunden dauern.
  • ISPs senden in der Regel einmal pro Tag, ca. 24 h nach UTC-Mitternacht.
  • Pollt deine Mailbox? Auf Mailboxen → jetzt prüfen.
  • Geht die Mail an die richtige Adresse? Mal selbst eine Mail an rua=… schicken — landet sie?
Upload-Status „ignored“.

Die im Report enthaltene Domain ist im Tenant nicht angelegt. Als Admin: Domain hinzufügen, dann erneut hochladen.

Polling-Intervall ändern.

In .env IMAP_POLL_INTERVAL_MINUTES=5 setzen, App neu starten. Sehr kurze Intervalle bringen wenig — ISPs schicken sowieso nur 1×/Tag.

Reverse-DNS macht Imports langsam.

RESOLVE_PTR=false in .env setzen. Bestehende Records bekommen damit keine Hosts mehr; künftige bleiben leer.

Wie melde ich mich als Superadmin um?

Logout → mit SUPERADMIN_EMAIL einloggen. Über Tenants in fremde Mandanten kannst du nicht hineinwechseln — das ist Absicht (Datentrennung). Wenn du als Superadmin Daten eines Mandanten editieren musst: dort einen User anlegen.

Wie reset ich das Admin-Passwort?

Es gibt absichtlich keine Selfservice-Recovery (kein SMTP gesetzt). Direkt in der DB: User-Eintrag löschen → ENVs setzen → App neu starten → Bootstrap legt einen neuen Superadmin an. Oder über die SQL-Konsole das password_hash-Feld mit einem neu erzeugten bcrypt-Hash überschreiben.

Postgres in der UI vermisst Daten von SQLite.

Es gibt bewusst keinen Auto-Migrationspfad. Für eine echte Migration: pg_loader oder ein Skript, das Reports per /api/v1/reports reimportiert.
