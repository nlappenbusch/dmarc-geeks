Self-Hosted Docker Auto-Deploy

Vom git push zum laufenden Container.
In 30 Minuten reproduzierbar.

Battle-tested Pipeline-Recipe: GitHub Actions Self-Hosted Runner trifft Docker-Compose-Stack auf einem internen Proxmox-LXC, vor dem ein Nginx Proxy Manager und Let's Encrypt steht. Du editierst lokal — Push fertig, Deploy automatisch.

TL;DR — 10 Befehle

Wenn du die Pipeline schon kennst und nur die Hand-On-Sequenz willst:

# Lokal
git init -b main && git add . && git commit -m "init"
git remote add origin https://github.com/USER/PROJECT.git
git push -u origin main

# Server (LXC/VM, Docker installiert, User 'actions' existiert)
sudo mkdir -p /opt/PROJECT && sudo chown actions:actions /opt/PROJECT
sudo -u actions git clone https://github.com/USER/PROJECT.git /opt/PROJECT
sudo -u actions git config --global --add safe.directory /opt/PROJECT
sudo chmod +x /opt/PROJECT/deploy.sh

# Runner (Tar aus benachbartem Runner kopiert oder fresh download)
cd /opt/actions-runner-PROJECT
sudo -u actions ./config.sh --url https://github.com/USER/PROJECT \
  --token TOKEN --name LABEL --labels LABEL --unattended --replace
sudo ./svc.sh install actions && sudo ./svc.sh start

Prerequisites

  • Lokal: Python 3.12+, Git, Docker (für lokale Tests).
  • GitHub: Account, leeres Repo (privat empfohlen).
  • Server (Proxmox LXC oder VM): Linux mit Docker + docker-compose-Plugin, User actions in der docker-Gruppe.
  • Reverse-Proxy: Nginx Proxy Manager (NPM) oder Caddy/Traefik auf separater VM mit Public-IP, Port 80/443 von außen erreichbar.
  • DNS: Zone bei einem Provider, A-Record auf NPM-Public-IP setzbar.

2

Pipeline-Files anlegen

deploy.sh (im Repo-Root, executable)

#!/usr/bin/env bash
set -euo pipefail
REPO_DIR="/opt/PROJECT"
cd "$REPO_DIR"

git fetch origin main
git reset --hard origin/main

# Compose-Vars aus der Shell raus (sonst koennen Reste aus anderen
# Runner-Jobs die .env ueberschreiben -- siehe Stolperfalle 2)
unset APP_PORT POSTGRES_USER POSTGRES_PASSWORD POSTGRES_DB \
      DATABASE_URL SECRET_KEY \
      # ... alle Variablen die in compose oder pydantic-Settings auftauchen

docker compose pull || true
docker compose build
docker compose down --remove-orphans
sleep 3
docker compose up -d

# Self-healing: in manchen LXCs bleibt Container in "Created"
sleep 3
if ! docker ps --filter name=PROJECT-app-1 --filter status=running -q | grep -q .; then
  docker start PROJECT-app-1 || true
  sleep 3
fi
if ! docker ps --filter name=PROJECT-app-1 --filter status=running -q | grep -q .; then
  echo "ERROR: app container not running"
  docker logs PROJECT-app-1 --tail=40 || true
  exit 1
fi

docker image prune -f

In Git als executable markieren — wichtig wenn du auf Windows entwickelst:

git update-index --add --chmod=+x deploy.sh

.github/workflows/deploy.yml

name: Deploy

on:
  push:
    branches: [main]
  workflow_dispatch:

concurrency:
  group: deploy-PROJECT
  cancel-in-progress: false

jobs:
  deploy:
    runs-on: [self-hosted, Linux, X64, LABEL]
    steps:
      - name: Run deploy script
        run: bash /opt/PROJECT/deploy.sh

docker-compose.yml-Pattern (Auszug)

services:
  app:
    build: .
    restart: unless-stopped
    env_file: [.env]
    ports:
      - "${APP_PORT:-8000}:8000"   # Host-Port aus .env
    volumes:
      - ./.env:/app/.env           # NUR wenn die App die .env zur Laufzeit
                                   # editieren koennen muss (z.B. Admin-Panel).
                                   # Beachte: rename() funktioniert auf einem
                                   # bind-mount-File nicht -- App muss
                                   # in-place schreiben koennen.
    depends_on:
      db: { condition: service_healthy }

  db:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    volumes:
      - data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER}"]
      interval: 5s
      timeout: 3s
      retries: 10

volumes:
  data:

3

Server-Setup (Docker-Host)

Auf der LXC/VM, einmalig:

sudo mkdir -p /opt/PROJECT
sudo chown actions:actions /opt/PROJECT
sudo -u actions git clone https://github.com/USER/PROJECT.git /opt/PROJECT
sudo -u actions git config --global --add safe.directory /opt/PROJECT
sudo chmod +x /opt/PROJECT/deploy.sh

Bei privaten Repos: PAT erstellen (GitHub → Settings → Developer settings → Personal access tokens → "repo"-Scope) und beim ersten git clone als Passwort eingeben. Git merkt's sich.

.env auf dem Server (NIE im Repo!)

cd /opt/PROJECT
[ -f .env ] && mv .env ".env.bak.$(date +%s)"

# Generieren mit openssl statt Python -- weniger Abhaengigkeiten
SECRET_KEY=$(openssl rand -base64 64 | tr -d '\n')
FERNET_KEY=$(openssl rand 32 | openssl base64 -A | tr '+/' '-_')
PG_PASS=$(openssl rand -base64 32 | tr -d '\n=+/' | head -c 32)
ADMIN_PASS=$(openssl rand -base64 24 | tr -d '/+=' | head -c 20)

cat > .env <<EOF
POSTGRES_USER=appuser
POSTGRES_PASSWORD=${PG_PASS}
POSTGRES_DB=appdb
APP_PORT=PORT
SECRET_KEY=${SECRET_KEY}
FERNET_KEY=${FERNET_KEY}
SUPERADMIN_EMAIL=admin@DOMAIN
SUPERADMIN_PASSWORD=${ADMIN_PASS}
BASE_URL=https://DOMAIN
COOKIE_SECURE=true
TRUSTED_PROXIES=true
EOF

chown actions:actions .env
chmod 600 .env

clear
echo "Login: admin@DOMAIN / ${ADMIN_PASS}"
# Kopier dir das ins Passwort-Manager, dann:
clear; history -c
Wichtig: nie SECRET_KEY/FERNET_KEY/Passwörter in den Chat oder ein Ticket pasten. Direkt im Terminal generieren, sofort in den PW-Manager, dann clear; history -c. Wenn doch geleaked: .env wegwerfen + neu generieren BEVOR ein User den ersten Login gemacht hat — danach würden gespeicherte verschlüsselte Werte den FERNET_KEY brauchen.
4

GitHub-Runner registrieren

Im GitHub-Repo: Settings → Actions → Runners → New self-hosted runner → Linux x64. Den Token aus der angezeigten ./config.sh ...-Zeile rauskopieren. Token läuft ~1h, also gleich weiter.

# Verzeichnis fuer diesen Runner
sudo mkdir -p /opt/actions-runner-PROJECT
sudo chown actions:actions /opt/actions-runner-PROJECT
cd /opt/actions-runner-PROJECT

# Tar aus benachbartem Runner kopieren spart Download
sudo -u actions cp /opt/actions-runner-OTHER/actions-runner-linux-x64-*.tar.gz . 2>/dev/null \
  || sudo -u actions curl -o runner.tar.gz -L \
       https://github.com/actions/runner/releases/download/v2.334.0/actions-runner-linux-x64-2.334.0.tar.gz

sudo -u actions tar xzf actions-runner-linux-x64-*.tar.gz

# Konfigurieren
sudo -u actions ./config.sh \
  --url https://github.com/USER/PROJECT \
  --token TOKEN_AUS_GITHUB \
  --name LABEL \
  --labels LABEL \
  --unattended --replace

# Service installieren + starten
sudo ./svc.sh install actions
sudo ./svc.sh start
systemctl status 'actions.runner.USER-PROJECT.LABEL.service' --no-pager

Auf GitHub im Runners-Tab muss LABEL jetzt grün und Idle stehen.

5

Erstes Deployment manuell ziehen

sudo -u actions bash /opt/PROJECT/deploy.sh
docker ps --filter name=PROJECT
curl -i http://127.0.0.1:PORT/healthz   # oder gleicher Endpoint deiner App

Beide Container Up, healthz 200 → Stack steht. Dann LAN-Test:

LAN_IP=$(ip -4 -o addr show | awk '$2 != "lo" {print $4}' | cut -d/ -f1 | head -n1)
curl -i http://$LAN_IP:PORT/healthz
Wenn LAN-Test fehlschlägt: Firewall (ufw/Proxmox-LXC-Firewall) blockt. ufw allow PORT/tcp oder im Proxmox-UI Container → Firewall → Rule hinzufügen.
6

NPM & DNS

1) DNS zuerst: A-Record DOMAIN auf Public-IP der NPM-VM. TTL 300s. Verify: nslookup DOMAIN.

2) NPM Proxy Host (Hosts → Proxy Hosts → Add):

  • Domain Names: DOMAIN
  • Scheme: http
  • Forward Hostname / IP: SERVER_LAN_IP
  • Forward Port: PORT
  • "Block Common Exploits" + "Websockets Support" ✓
  • SSL-Tab: "Request a new SSL Certificate" + "Force SSL" + "HTTP/2" + LE ToS ✓

Verify vom PC: curl.exe -I https://DOMAIN → HTTP/2 200.

7

Pipeline live testen

Lokal eine triviale Änderung:

git commit --allow-empty -m "test deploy"
git push

Auf GitHub Actions-Tab: Workflow muss starten und nach 30–90s grün werden. Auf dem Server:

sudo -u actions git -C /opt/PROJECT log -1 --oneline
journalctl -u actions.runner.USER-PROJECT.LABEL --no-pager -n 4 | tail -2
Wenn der Job grün läuft: Pipeline ist live. Ab jetzt deployt jeder git push auf main automatisch.

Stolperfallen & Fixes

Aus echtem Schmerz extrahiert. Wenn du auf einen davon stößt, ist die Lösung 30 Sekunden weg.

Port-Kollision

Anderer Container bindet schon den Port. Compose-up scheitert mit "port is already allocated".

→ Anderen APP_PORT in .env, NPM-Forward anpassen.

Shell-Env überschreibt .env

Compose nimmt Shell-Env vor .env-Datei. Wenn der Runner-Shell APP_PORT=ALT hat, bleibt der falsche Port aktiv.

→ unset APP_PORT POSTGRES_* am Anfang von deploy.sh.

Container in "Created", nicht "Up"

Compose-up schließt mit Erfolg ab, aber Container startet nicht (Race-Condition mit docker-proxy auf manchen LXCs).

→ Self-healing-Block in deploy.sh: docker start als Fallback.

NAT-Loopback zu eigenem Mailserver

Container will mail.firma.ch erreichen, DNS löst auf Public-IP, Router routed nicht zurück.

→ extra_hosts in compose mit FQDN→LAN-IP, oder SMTP_HOST = LAN-IP + SMTP_TLS_VERIFY=false.

nft DOCKER-Chain hat kein DNAT für den neuen Port

Compose-up scheinbar ok, aber externe Verbindung läuft ins Leere.

→ docker compose down + sleep 3 + up -d; im Notfall systemctl restart docker.

Bind-mounted .env, atomic-rename fail

Wenn die App .env editierbar braucht: Linux verbietet rename() auf eine bind-gemountete Single-File. os.replace wirft OSError.

→ App muss in-place schreiben (Schreiben in temp + Fallback auf p.write_text()).

YouTube/iframe-Embed Error 153

Wenn deine App YouTube-Embeds zeigt und Player-Config-Error wirft.

→ www.youtube.com/embed statt -nocookie; modestbranding-Param raus; CSP frame-src erweitern.

Pydantic crasht weil Number-Field leer

Admin-Panel speichert leeren String in .env, beim nächsten get_settings() wirft Pydantic "Input should be a valid integer".

→ Im Save-Handler leere number/url/email-Felder skippen, nicht als "" persistieren.

Hardening (sobald produktiv)

  • Postgres-Backup automatisieren: docker exec ... pg_dump via cron, off-site verschlüsselt.
  • Container-Updates: Renovate/Dependabot für requirements.txt + docker-compose.yml.
  • Read-only-Container: read_only: true in compose, Schreib-Pfade explizit als tmpfs/Volume.
  • Dockerfile-Healthcheck: HEALTHCHECK --interval=30s CMD curl -f http://localhost:8000/healthz.
  • Secrets-Rotation: SECRET_KEY/SMTP-Passwort in 6-Monats-Rhythmus.
  • NPM-Backups: SSL-Configs + Forward-Hosts gehen sonst verloren wenn die NPM-VM stirbt.
  • Monitor: simpler Cron auf dem Host: curl -sf http://127.0.0.1:PORT/healthz || alert.
  • Audit-Log auf der App: Login/Logout, Settings-Änderungen, Privilegierte Aktionen.
